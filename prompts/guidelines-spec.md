Specification guidelines:
- Treat SPEC as an optional preflight step for turning the user's request into testable behavior.
- Translate the request into a user story, concrete acceptance criteria, and the tests that will prove them.
- Start at the outermost seam that honestly proves the request. Route, API, redirect, page, and form requests should start there rather than in helpers, services, schema, or migrations.
- If the request is entirely schema, migration, database, or setup work, ask which business behavior it supports before treating it as a standalone feature slice.
- Choose one cheapest honest first proof level for each item: unit for isolated logic, integration for boundaries and contracts.
- Keep spec items observable and tied to user-visible behavior rather than implementation details. Spec items like "schema has N columns" or "config value equals X" describe static declarations, not behavior — reframe them as the behavior the declaration enables.
- Boundary and contract behavior should usually name integration proof explicitly.
- Prefer small vertical slices that prove the user-visible behavior end to end before support work.
- Helper, schema, migration, and service checks are support work unless the user explicitly asked for internals.
- Tighten vague, overlapping, or untestable items before moving into RED.
